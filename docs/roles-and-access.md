# Roles & Access Control

ClawFarm has session-based authentication with two roles: **admin** and **user**. Access to individual bots is controlled per-user via the `bots` field.

## Roles

### Admin

Full system access. Can manage all bots regardless of their `bots` field, and has exclusive access to user management.

### User

Standard role. Can only interact with bots they've been explicitly granted access to (or all bots if assigned `*`).

## Bot Access (`bots` field)

Each user has a `bots` list that controls which bots they can see and interact with:

| Value | Meaning |
|-------|---------|
| `["*"]` | Access to all bots (current and future) |
| `["bot-a", "bot-b"]` | Access to only the named bots |
| `[]` | No bot access |

For admin users, the `bots` field is ignored — admins always have implicit access to every bot.

## Permission Matrix

| Action | Admin | User (`*`) | User (specific bots) | User (no bots) |
|--------|:-----:|:----------:|:--------------------:|:--------------:|
| View bot list | All | All | Assigned only | Empty |
| Create a new bot | Yes | Yes | Yes | Yes |
| Start / stop / restart bot | Any | Any | Assigned only | None |
| Delete bot | Any | Any | Assigned only | None |
| View bot logs | Any | Any | Assigned only | None |
| Duplicate / fork bot | Any | Any | Assigned only | None |
| Backup / rollback bot | Any | Any | Assigned only | None |
| Open bot Control UI | Any | Any | Assigned only | None |
| View fleet stats | Yes | Yes | Yes | Yes |
| View templates | Yes | Yes | Yes | Yes |
| Change own password | Yes | Yes | Yes | Yes |
| List all users | Yes | No | No | No |
| Create / edit / delete users | Yes | No | No | No |

## Auto-Grant on Bot Creation

When a non-admin user creates, duplicates, or forks a bot, the new bot is automatically added to their `bots` list. This means any user can create a bot and immediately manage it, even if they started with an empty `bots` list.

Admins are skipped during auto-grant since they already have implicit access to everything.

## How Access Is Enforced

Access control is enforced at two layers:

1. **Caddy (reverse proxy)** — Before a request reaches a bot's Control UI (`/claw/{name}/`), Caddy makes a forward-auth subrequest to the dashboard. If the user doesn't have access to that bot, they get a 403 before the request ever reaches the bot container.

2. **FastAPI (API)** — Every bot-specific API endpoint (`/api/bots/{name}/...`) checks the user's session and bot access. Unauthorized requests return 403.

The bot list endpoint (`GET /api/bots`) is also filtered server-side — non-admin users without `*` access only receive bots they're assigned to. They cannot see that other bots exist.

## First Run & Bootstrap

On first startup, if no users exist, ClawFarm creates a default admin account:

- **Username**: value of `ADMIN_USER` env var (default: `admin`)
- **Password**: value of `ADMIN_PASSWORD` env var. If unset, a random password is generated and printed to stdout with a prominent banner.

```env
# .env
ADMIN_USER=admin
ADMIN_PASSWORD=your-secure-password
```

## Sessions

- Sessions are stored in memory and **lost on dashboard restart** — all users must re-login.
- Default session lifetime is 24 hours (configurable via `SESSION_TTL`).
- RBAC changes (role, bot assignments) take effect **immediately** — the dashboard re-reads the user store on every request. No re-login required.
- When a user account is updated or deleted, all their active sessions are invalidated instantly.
- Login is rate-limited to 5 attempts per IP in a 5-minute window.

## Disabling Authentication

Set `AUTH_DISABLED=1` (or `true`) to skip all authentication. Every request is treated as a full admin with access to all bots. Caddy injects a hardcoded `X-Forwarded-User: dev` header.

Listing users returns an empty list; creating, updating, or deleting users returns a 400 error.

```env
AUTH_DISABLED=1
```

## Managing Users

Admins manage users from the **Users** page in the dashboard (header link). From there you can:

- Create new users with a username, password, role, and bot access list
- Edit existing users (change role, password, or bot assignments)
- Delete users (you cannot delete yourself or the last admin)
- Demoting the last admin is also prevented

Non-admin users see an **Account** page instead, where they can only change their own password.

## User Store

Users are stored in `bots/.users.json` (persists via the existing volume mount). The file is a JSON object keyed by username:

```json
{
  "admin": {
    "password_hash": "$2b$12$...",
    "role": "admin",
    "bots": ["*"]
  },
  "alice": {
    "password_hash": "$2b$12$...",
    "role": "user",
    "bots": ["research-bot", "writing-bot"]
  }
}
```

Passwords are hashed with bcrypt (12 rounds). Do not edit this file manually unless you know what you're doing — use the dashboard UI or API instead.

## Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `ADMIN_USER` | `admin` | Bootstrap admin username |
| `ADMIN_PASSWORD` | (random) | Bootstrap admin password. Auto-generated if unset |
| `SESSION_TTL` | `86400` | Session lifetime in seconds (24h) |
| `AUTH_DISABLED` | (unset) | Set to `1` or `true` to disable auth entirely |
